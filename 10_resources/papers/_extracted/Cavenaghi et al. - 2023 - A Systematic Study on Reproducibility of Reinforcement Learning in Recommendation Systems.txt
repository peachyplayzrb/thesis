As reported in [58] and [39], traditional techniques implementing RSs involve content-based filter- ing [52], collaborative-filtering [57], knowledge-based [7], and hybrid methods [6]. Nonetheless, as stated in [66] and [38], these methods suffer from many problems such as cold start, scalabil- ity,low-qualityrecommendations,andhighcomputationaldemand.Othermethodologies,likethe one presented in [30], exploitDeep Learning(DL)[ 25], a framework based on artificial neural networkswithseveralstackedlayers.DLgainedpopularityinRSs,duetohisabilitytolearncom- plexrelationshipshidden in thehistoricaldataand thusprovide more effective recommendations. One example of a successful application to RSs is provided by the GRU4REC method presented in [32]. However, DL methods are usually non-interpretable and require a lot of data to work well, thus, their superiority over traditional techniques in RSs is still under debate [21,56]. However, an RS is not asked to solve a prediction problem, but a sequential decision-making problem since we have to actively recommend items to users. This allows the recommendation problem to be modelled as aMulti-Armed Bandit problem(MAB)[ 59]o r Markov Decision Process (MDP)[ 5], and thus to be solved withReinforcement Learning(RL)[ 64]. MAB is a powerful framework that allows agents to solve sequential decision-making problems under uncertainty [60]. In the standard version, an algorithm hasK possible actions (or arms) to choose from andT rounds (or timesteps). In each round, the algorithm chooses an arm, performs the associated action, and receives a reward for its choice. The reward is drawn independently from some fixed but unknown probability distribution, which depends only on the chosen arm. Thus, aftereachround,thealgorithmobservestherewardforthechosenarm,butnotfortheotherarms. AnextensionoftheMABframeworkistheMDPframework,since,inMDPs,pastchoicesinfluence thecurrentstateandthustheactionswecantakeandtherewardswecanobtain.Formally,MDPs aredefinedasatuple (S,A,P,R,γ)whereSisthesetofstates, Aisthesetofactions, P :S×A→ S is the transition probabilityP (st +1 =s′|st =s,at =a), i.e., the probability of the states′ at time t + 1 given that the agent performs actiona in states at timet, R :S×A→ R is the reward function defined asR(st = s,at = a) = E[rt|st = s,at = a]w h i l eγis a discount factor. The RS problemcanbemappedtoMDPsinseveralways.Forinstance,byassociatingstateswiththeusers’ representation, actions to the recommendation of each item, and rewards to the user’s feedback that we are interested in. For a complete introduction to RL in RSs refer to [2]. The goal of an RS algorithm is to maximize user satisfaction, which can be translated into maximizing the reward signal, or multiple reward signals like in [61]. Given the ability of RL algorithms to handle sequential decision problems, learn by a trial-and- error process and focus on delayed rewards, there are several RL approaches used in RSs. Specif- ically, the last aspect is particularly relevant in RSs, where a myopic approach, which disregards long-term effects, can lead to sub-optimal results [29]. Some approaches use basic RL algorithms suchasQ-Learning[ 77,91],MonteCarloTreeSearch[ 90],Tree-structuredPolicyGradient[ 9],and REINFORCE with baseline [74], while others use DL with RL like Q-Learning with aGenerative Adversarial Network(GAN)[ 14] or actor-critic methods [77, 78]. In recent years, several papers, published in Science [35], AAAI [27], and RecSys [21], have investigated the aspect of replication and reproduction of recent AI and Machine Learning research. [27], for instance, quantifies the state of reproducibility of empirical AI research by analysing a total of 400 research papers from the conference series IJCAI and AAAI. The work of [21] focuses on the DL application on RSs. One main reason behind the observed replicability and reproducibility problem is the lack of open-source artifacts, i.e., the authors rarely share the code used to perform experiments. Although the number of published artifacts has increased in recent years [27], it still remains rare to have access to them. In this section, we want to present some good practices to improve reproducibility, following the ideas proposed in [31] and [54]. Fundamentals. The fundamental artifacts that must be shared to enable the reproducibility of the experiments are (i) the raw form of the datasets, (ii) the pre-processing code, (iii) the algo- rithm implementation, and (iv) the code to perform the experiments. Furthermore, it is neces- sary to provide all these artifacts in a “publicly accessible archival repository” such asFigshare (https://figshare.com/), Dryad (https://datadryad.org/), orZenodo (https://zenodo.org). In this ar- ticle, we consider valid also GitHub personal repositories, but these are not recognized by ACM since “personal web pages are not acceptable for this purpose”. In case the dataset is already pub- licly available, the authors should indicate the exact source from which they downloaded the ver- sionofthedatasetusedfortheexperiment,whileinthecaseofaprivatedataset,itshouldbemade publiclyavailableinrawform.However,werecognizethatnotalldatasetscanbemadepublicfor reasons of privacy or data use agreements. The solution to this problem may be different for each specific case, but authors should try to find the best tradeoff between privacy and reproducibility to improve confidence in their results. Important. Whiletherecommendationsinthepreviousparagrapharefundamentaltoenablere- producibility,theyarenotenoughtoensureexactreproducibility.Severalanalysedpapersfulfilled the above recommendations, but we still obtained different results on the same experiments, per- formed with the same code on the same datasets. These differences can result from many factors, e.g., different versions of third-party software and hardware characteristics. To this end, the use of dependency management tools, such asPackrat (https://rstudio.github.io/packrat/)a n dConda (https://conda.io/), or containerization, usingDocker (https://www.docker.com/), can help to re- solve this problem. Regarding the hardware, it is good practice to report the main characteristics of the hardware used to perform the experiments, so that a third-party analyst can take any dif- ferences into account. Among the main information there are the operating system version, the model and number ofCentral Processing Units(CPUs) and GPUs, and the amount of CPU RAM and GPU RAM required. Furthermore, the authors should also share the hyperparameters of the proposed algorithm and the baselines, directly in the paper or at least within the artifacts, sincetheseheavilyinfluencetheperformanceofthealgorithms.Finally,randomizationisanother key aspect that determines the results of the experiments in machine learning analyses. Indeed, it is good practice to set and share the seed of the pseudorandom number generators to ensure reproducible results. Additionals. In the previous paragraphs, the authors should provide all the steps the authors took to use the data, from the raw data to the input given to the algorithm, and the necessary documentationtousethem.However,tofacilitateotherresearcherstoreproducetheexperiments, the entire analysis (including downloading data, pre-processing data, training models, producing output, and generating the results shown in the paper) should be reproducible with a single com- mand. A simple shell script can accomplish this goal, but workflow management software, like Snakemake (https://snakemake.readthedocs.io/en/stable/) or Nextflow (https://www.nextflow.io/) providesseveraladvantageswithoutextraworkfromtheauthors.Finally,authorsshouldalsostore andpublishintermediateoutputstoallowreproducibilityevenincasethealgorithmrequireshuge computational resources and time.
